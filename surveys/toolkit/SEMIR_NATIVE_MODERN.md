# SemIR Native Modern Toolkit Sweep

Last swept: 2026-05-30.

This note tracks the Action Toolkit as integration pressure for the
SemIR-native backend under the `modern` profile. The compatibility backend
already compiles the Toolkit; this pass is about identifying source idioms that
need modern spelling and backend shapes that still need structured support.

## Source Set

Use original Toolkit sources unless the old source relies on a compatibility
idiom that the modern profile intentionally rejects. Modern-specific copies
live in `samples/toolkit/modern`; the directory name carries the profile
meaning, so filenames match the original Toolkit names.

Modern copies created so far:

| Modern source | Original source | Reason |
| --- | --- | --- |
| `ALLOCATE.ACT` | `ALLOCATE.ACT` | Defines the required `EndProg` harness symbol and desugars record-field compound assignments. |
| `PMG.ACT` | `PMG.ACT` | Uses `@GraphP0` for `Zero`, stages complex address/loop expressions, and spells pointer-like locals as pointers. |
| `PRINTF.ACT` | `PRINTF.ACT` | Replaces the loose variadic argument pointer with explicit `CARD POINTER` / `INT POINTER` cursors and a cast. |
| `REAL.ACT` | `REAL.ACT` | Replaces assignment to a routine name with a direct `ROM_IFP()` call. |
| `SORT.ACT` | `SORT.ACT` | Replaces routine-name assignment dispatch with direct dispatcher routines. |

These copies are source-level modernization shims, not backend policy. Keep
new copies narrow: change only old-profile idioms or small expression staging
needed to expose the next native backend shape.

## Sweep Commands

Run only the modern copies:

```sh
cargo run --bin actionc-semir-sweep -- \
  --candidate native \
  --profile modern \
  --validation coverage \
  samples/toolkit/modern
```

Run the mixed Toolkit set, using original sources for files that do not need a
modern copy:

Comparison interpretation:

- These are backend-to-backend deltas, not original compiler parity results.
  Use `TOOLKIT_COMPARE.md` for original VM capture comparisons.
- Run-address drift is expected while SemIR-native emits different routine
  prologues, storage layout, and branch veneers. It is still useful as a quick
  signal for which files have large layout changes.
- First-byte diffs at offset `0` or near the first routine usually mean storage
  or trampoline layout differs before substantive statement code is reached.
- The highest-value next comparisons are instruction-level checks for
  `SORT.ACT`, `PRINTF.ACT`, and `PMG.ACT`; all three now
  compile, so work should focus on whether the native shape is merely
  conservative or materially worse.

Runway policy for these deltas:

- Do not pursue broad SemIR-native size optimization just to close the +35%
  aggregate delta.
- Keep exactness work limited to semantic correctness, runtime behavior, and
  clear layer-boundary cleanup.
- Preserve these measurements as TAC input, especially around branch veneers,
  routine layout, and repeated materialization costs.

## Backend Backlog From Toolkit

No current Toolkit item is blocked on a missing SemIR-native backend shape.
Future work should move from "make it compile" to comparison triage:

| Area | Follow-up |
| --- | --- |
| Routine entry/run-address deltas | Compare source maps and routine boundaries before deciding whether these are acceptable SemIR-native layout differences. |
| Code-size deltas | `tac-deferred`: measure conservative long-branch veneers and broader materialization choices, then carry the data into TAC planning. |
| Byte-level deltas | Use `actionc-compare` on one file at a time; prefer focused backend tests before changing shared lowering. |

Recently closed Toolkit-native shape gaps:

- Fixed-address user routines such as `PROC CIO=$E456(...)` and the REAL ROM
  routines are registered as absolute call targets.
- Predefined builtin variables `color` (`$02FD`) and `device` (`$00B7`) resolve
  to native storage aliases.
- Local `TYPE` / `RECORD` declarations register record layout without emitting
  storage.
- Statement-level `DEFINE` declarations no-op at declaration emission time.
- Word runtime compound operators (`*=`, `/=`, `MOD=`) use the Action runtime
  helpers, closing `TURTLE.ACT`'s `TG_Phi==MOD 360` gap.
- Word array indexes accept word expressions and use the low byte as the
  dynamic index, covering `TURTLE.ACT` sine table lookups.
- Byte assignments and byte arithmetic accept the low byte of word runtime
  results, word pointer dereferences, word arrays, and word locals where the
  Toolkit uses Action!'s truncating behavior.
- Dynamic byte pointer indexed compounds materialize the indexed address once
  and preserve it across the read-modify-write, covering the `PMG.ACT`
  mask update.
- Byte `LSH` / `RSH` expressions accept computed operands via runtime helpers,
  covering `PMG.ACT` width and collision helpers.
- Conditional branch emission uses absolute-jump veneers when the target is
  forward/unbound or already known to be out of relative range, covering larger
  Toolkit routines and loops.
- Define-backed no-parens macro calls such as `SaveTemps` / `GetTemps` parse
  their `DEFINE` value as `MachineItem`s and reuse native machine-block
  emission, covering `CONSOLE.ACT`.

## Guardrails

- Do not port AST codegen shapes wholesale. Use Toolkit failures as examples,
  then add the smallest SemIR-native shape in the owning layer.
- Prefer source copies only for old Action! idioms that modern deliberately
  makes explicit: routine-name assignment, loose pointer reinterpretation,
  and required harness symbols.
- Do not hide backend gaps with broad source rewrites. If the rewritten source
  is less representative of Toolkit behavior, keep the failure and document the
  backend shape.
- When a Toolkit failure maps to a stress backlog item, add or reuse a focused
  stress program first, then rerun the mixed Toolkit sweep.
- Keep compatibility survey files separate from this note. `TOOLKIT_SURVEY.md`
  and `TOOLKIT_COMPARE.md` remain the AST/compat baseline.
