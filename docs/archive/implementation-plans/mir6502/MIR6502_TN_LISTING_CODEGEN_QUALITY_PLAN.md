# MIR6502 TN Listing Codegen Quality Plan

Snapshot date: 2026-06-07.

This is a Codex-ready implementation note derived from a compiler-generated TN
source listing. It complements, rather than replaces:

- `docs/MIR6502_CODEGEN_QUALITY_FIRST_SLICES_PLAN.md`
- `docs/MIR6502_RISK_AVERSE_CODE_QUALITY_PLAN.md`
- `docs/MODERN_OPTIMIZATION_ROADMAP.md`
- `docs/ACTION_COMPILER_VM_USAGE.md`

The note is intentionally split into small parts. Give Codex one part at a time.
Do not ask it to implement the whole roadmap in a single patch.

## Listing Diagnosis

The TN listing shows broadly correct compiler output, but the generated code is
still dominated by compiler-temporary memory traffic and conservative control-flow
lowering.

Observed baseline from the TN listing used for this note:

```text
PROC blocks:                       105
real instruction count:            about 7,088 instructions
real code bytes:                   about 17,789 bytes
LDA instructions:                  2,394
STA instructions:                  1,974
LDA+STA share of instructions:     about 61.6%
LDA+STA share of code bytes:       about 65.0%
spill data labels:                 879
STA m; LDA m adjacent pairs:       216
absolute accesses to $00xx:        322
branch-over-JMP, branchable:       159
LDA x; CMP #$00; BEQ/BNE:          57
JSR f; RTS tail-call sites:        19
constant negation emitted at run time: 7
x = x +/- 1 via load/add/store:    64
```

Treat these numbers as diagnostic baselines, not as permanent test assertions.
They will shift as fixtures, layout, and listing rendering improve.

The highest-level problem is not one missed peephole. The backend still
materializes many intermediate values into static spill slots and immediately
reloads them. Fixes should therefore proceed from safe observability, to local
peepholes, to producer-consumer materialization, and only then to liveness-aware
allocation.

## Global Guardrails

Apply these to every part:

- Keep MIR6502 changes behind existing configuration where practical, especially
  `Mir6502Config::enable_peepholes`.
- Run MIR verification before and after any MIR-transforming pass.
- Do not cross calls, machine blocks, pointer writes, OS/runtime calls, hardware
  registers, unknown absolute memory, or label joins unless effects/proofs are
  explicit.
- Do not make emission recover source semantics. Use MIR/NIR facts.
- Do not add source-name-specific rewrites for TN.
- Prefer fixture-level byte/listing assertions for each pattern before enabling a
  transformation broadly.
- If a rewrite changes flags or depends on flags being dead, require an explicit
  proof or keep it disabled.

## Part 2: Safe Addressing And Adjacent Reload Cleanup

### Goal

Remove mechanically safe local waste without changing MIR semantics or doing
global optimization.

### Candidate rewrites

Start with patterns that are either emitter-selection fixes or same-basic-block
local rewrites.

#### Zero-page opcode selection

Current listings include absolute direct accesses to zero-page addresses:

```asm
LDA $00E0
STA $00E0
```

When the operation has a legal zero-page opcode, emit:

```asm
LDA $E0
STA $E0
```

This is not a semantic optimization. It is target addressing-mode selection.

Scope:

- `LDA`, `LDX`, `LDY`
- `STA`, `STX`, `STY`
- `ADC`, `SBC`, `AND`, `ORA`, `EOR`, `CMP`
- `INC`, `DEC`
- only direct `$00xx` operands, not indexed/indirect forms unless the emitter
  already has the exact legal zero-page mode

#### Adjacent `STA m; LDA m`

Current example from `Putchar`:

```asm
LDA $2E68
STA $2E6B
LDA $2E6B
JSR Internal
```

The reload is unnecessary when `A` is still known to contain the stored value and
there is no barrier between the store and reload.

Do not remove the `STA` unless a separate liveness/dead-store proof says the
memory write is dead. In this part, remove only the redundant reload.

#### `LDA x; CMP #$00; BEQ/BNE`

`LDA` already sets Z/N. If the branch only needs Z and no instruction between
the load and branch changes flags, remove `CMP #$00`.

This is safe for `BEQ` and `BNE`. Do not generalize to signed or carry-sensitive
branches.

#### Constant negation

Fold:

```asm
LDA #c
EOR #$FF
CLC
ADC #$01
```

to:

```asm
LDA #(-c & $FF)
```

only when the final flags produced by the replacement are acceptable. If flag
equivalence is not modeled, restrict this to cases where flags are dead before
the next flag consumer.

### Acceptance

- Existing MIR6502 fixtures remain green.
- Add focused byte-level tests for:
  - zero-page direct `LDA`/`STA` selection;
  - adjacent `STA m; LDA m` reload removal with no intervening barrier;
  - no reload removal across a call, label join, machine block, or unknown memory
    mutation;
  - `LDA; CMP #0; BEQ/BNE` cleanup;
  - constant negation fold or an explicit deferred test if flag liveness is not
    ready.
- The listing-quality report from Part 1 shows reductions in the relevant counts.

### Suggested Codex task

```text
Implement MIR6502 safe local code-quality Slice 1.

Goal:
- Select zero-page opcodes for direct $00xx accesses where legal.
- Remove adjacent redundant LDA reloads after STA when tracked state proves A
  still contains the stored value.
- Remove CMP #$00 after LDA for BEQ/BNE when no flag-clobbering instruction is
  between them.
- Keep all changes local and guarded.

Do not implement:
- dead-store elimination;
- global copy propagation;
- cross-block value forwarding;
- branch-over-JMP rewriting;
- INC/DEC replacement;
- liveness-aware spill allocation.

Acceptance:
- Focused tests cover positive and barrier-negative cases.
- `cargo test -q mir6502 --lib` passes.
- `cargo test -q mir6502_fixtures_match_snapshots` passes or fixture updates are
  justified as intentional code-quality changes.
- The listing-quality report shows fewer $00xx absolute accesses, adjacent
  STA/LDA reloads, and CMP #0 after LDA patterns.

Suggested commit:
- mir6502: add safe local codegen peepholes
```

## Part 3: Branch And Tail-Call Cleanup

### Goal

Reduce control-flow scaffolding once code-shape metrics and local peepholes are
in place.

### Candidate rewrites

#### Tail call

Replace:

```asm
JSR target
RTS
```

with:

```asm
JMP target
```

only at routine tail position, when no cleanup code is required after the call.

#### Branch over JMP

Replace:

```asm
Bcc skip
JMP target
skip:
```

with:

```asm
B!cc target
```

only when `target` is reachable by a relative branch from the rewritten branch
site. Keep non-branchable far targets unchanged.

#### Jump-only block threading

At MIR level, remove or thread blocks whose only effect is `JMP target`, provided
the transformation preserves labels required by maps/listings and does not cross
unknown boundaries.

### Implementation notes

Prefer doing branch cleanup at MIR terminator/block-layout level before emission.
Avoid byte-patching raw emitted code as the primary implementation strategy.

If branch cleanup changes block ordering or label placement, update source
listing expectations deliberately.

### Acceptance

- Tail-call tests prove `JSR; RTS` becomes `JMP` only at routine tail.
- Branch inversion tests cover:
  - branchable target;
  - non-branchable target left unchanged;
  - label/mapping preservation;
  - branch condition inversion table.
- Listing-quality report shows fewer branchable branch-over-JMP patterns and
  fewer tail-call sites.
- No VM/runtime behavior changes in TN startup smoke checks.

### Suggested Codex task

```text
Implement MIR6502 control-flow cleanup Slice 2.

Goal:
- Convert tail-position JSR+RTS to JMP.
- Invert branch-over-JMP only when the JMP target is within relative branch range.
- Thread jump-only MIR blocks where labels/maps remain valid.
- Add focused tests for each rewrite and negative case.

Do not implement:
- full block layout optimization;
- profile-guided layout;
- far-branch expansion;
- global jump table generation;
- data/code rendering changes.

Acceptance:
- Branch inversion is range-checked.
- Tail-call conversion only applies at true routine tail.
- Existing source maps/listings stay deterministic.
- MIR verification passes before and after the cleanup.
- TN listing metrics improve for tail calls and branch-over-JMP patterns.

Suggested commit:
- mir6502: clean up tail calls and branch-over-jmp
```

## Part 4: Producer-Consumer Materialization And Spill Reduction

### Goal

Stop creating short-lived spill traffic in the first place for simple
producer-consumer chains.

This part overlaps with the existing first-slices plan. Complete or respect those
slices before broadening this work.

### Candidate producer-consumer folds

Handle one use-def shape at a time:

```text
Store(dst, constant byte/word)
Store(dst, Load(src))
Store(dst, CallResult)
Store(dst, unary/binary result) when the operation already leaves the value in A
Return(CallResult)
Branch(CompareResult) where flags can be consumed directly
```

The key rule: if a producer's only consumer immediately stores, returns, or
branches on the value, materialize directly into the consumer path instead of
assigning an ordinary spill home.

### Spill slot accounting

Before eliminating spills, add accounting:

```text
spills allocated
spills emitted into storage
spills written
spills read
spills with exactly one write and one immediate read
spills live across calls
spills live across block joins
```

Use this to pick safe folds. Do not try to solve all spills in one patch.

### Acceptance

- `Putchar`-like sequences no longer copy through a one-use spill before a call
  when the accumulator already holds the argument.
- Constant pointer/value assignment avoids:
  `LDA #lo; STA spill; LDA #hi; STA spill+1; LDA spill; STA dst ...`
- Call-result-to-store cases use `$A0/$A1` directly when the result has one
  immediate store consumer.
- Multi-use values and values crossing calls/block joins still get materialized.
- Spill counts in the listing-quality report decrease without correctness
  regressions.

### Suggested Codex task

```text
Implement MIR6502 producer-consumer materialization Slice 3.

Goal:
- Add spill accounting to identify one-write/one-read short-lived spills.
- Implement one narrow direct materialization path, preferably Store(dst,
  CallResult) or Store(dst, Load(src)), using existing MIR facts.
- Add focused tests proving direct materialization and negative cases.

Do not implement:
- global copy propagation;
- dead-store elimination;
- alias-sensitive forwarding;
- register allocation;
- spill-slot coloring;
- cross-block value numbering.

Acceptance:
- The selected one-use producer-consumer pattern no longer allocates or reloads an
  ordinary spill.
- Multi-use and cross-barrier values still materialize safely.
- MIR verification remains strict.
- Fixture/listing changes are documented as intentional code-quality changes.
- Listing-quality metrics show lower spill/reload counts for the covered pattern.

Suggested commit:
- mir6502: materialize one-use values into consumers
```

## Part 5: Increment/Decrement And Flag-Sensitive Arithmetic Idioms

### Goal

Replace obvious `x = x +/- 1` byte updates with `INC`/`DEC` only when flags and
accumulator value requirements are understood.

### Candidate rewrite

```asm
LDA m
CLC
ADC #$01
STA m
```

to:

```asm
INC m
```

and:

```asm
LDA m
SEC
SBC #$01
STA m
```

to:

```asm
DEC m
```

This rewrite is not always flag-equivalent:

- `INC`/`DEC` do not affect carry.
- the original sequence leaves the new value in `A`; `INC`/`DEC` do not.
- Z/N behavior is comparable for the updated byte, but carry and A availability
  differ.

Therefore require:

```text
carry is dead
A value is dead
only Z/N are required, or all flags are dead
memory address is direct and legal for INC/DEC
```

### Acceptance

- Positive tests cover `carry dead` and `A dead`.
- Negative tests cover:
  - carry consumed after the update;
  - A consumed after the update;
  - volatile/hardware/unknown memory;
  - indexed/indirect forms without legal instruction support.
- Listing-quality metrics show fewer load/add-one/store and load/sub-one/store
  patterns.

### Suggested Codex task

```text
Implement MIR6502 flag-safe INC/DEC peephole Slice 4.

Goal:
- Replace byte x=x+1 and x=x-1 load/add/store sequences with INC/DEC only when
  carry and A are proven dead and the memory mode is legal.
- Add focused positive and negative tests.

Do not implement:
- word increment/decrement;
- loop induction-variable optimization;
- cross-block liveness;
- indexed-address rewrites;
- broad arithmetic simplification.

Acceptance:
- The rewrite is disabled unless the required flag/A liveness proof exists.
- Tests prove carry-sensitive and A-sensitive cases are preserved.
- MIR6502 fixtures remain green.
- Listing-quality report shows fewer +/-1 load/store patterns.

Suggested commit:
- mir6502: use inc dec for flag-safe byte updates
```

## Part 6: Later Architecture Work, Not First Codex Tasks

Do not give these to Codex until the earlier parts are done and measured.

### Liveness-aware temporary allocation

Replace one-spill-per-temp behavior with:

```text
A/X/Y value tracking
zero-page scratch allocation
liveness-based materialization
spill-slot coloring
call-boundary preservation rules
routine-local spill overlaying when recursion/reentrancy rules permit
```

This needs strong use-def, dominance, call effects, and memory effects.

### Indexed addressing and array idioms

Use range/index proofs to select direct 6502 addressing modes:

```text
byte array + byte index -> absolute,Y or (zp),Y
word array + bounded index -> scaled index in Y when safe
constant element index -> direct byte offset
prepared pointer + known Y -> reuse Y when still valid
```

This should build on existing index-address proofs and the materialization work,
not bypass it in emission.

### Branch layout and jump tables

Large dispatch routines may eventually benefit from block layout or jump tables.
Do not start here. First remove local branch/JMP scaffolding and make metrics
visible.
