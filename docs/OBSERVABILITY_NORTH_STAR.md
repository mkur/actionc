# Observability North Star

The current "telescope" work should stay useful without becoming a tooling
cul-de-sac.

## Layer 1: Map Query

`CodegenMap` plus a small query layer is the low-level observability foundation.
It answers physical questions:

- which routine covers this generated address?
- which storage symbol owns this byte?
- which source range emitted this code?
- which skipped range reserved this space?
- which map items overlap this generated range?

This layer is valuable for compatibility work, profile comparison, layout
debugging, and explaining Action!-style codegen decisions.

Do not overbuild this layer into an interactive system too early. A CLI and JSON
output are enough until the query vocabulary proves itself in daily comparison
work.

## Layer 2: Provenance

If the compiler moves into deeper optimization, address ownership alone will not
be expressive enough. Inlining, global optimization, constant folding, dead-code
elimination, storage coalescing, and register allocation can all make one
generated byte belong to several logical causes.

The future-facing abstraction should be provenance:

- primary source span
- optional provenance stack, such as callsite -> inlined routine -> expression
- symbol references involved in the generated byte range
- transform tags, such as `inlined`, `constant-folded`, `dead-store-removed`,
  `register-reused`, or `storage-coalesced`
- stable IR or operation ids, if/when an IR exists
- physical placement, such as code address, storage address, register, or
  zero-page scratch location

At that point the useful question becomes not only "who owns address `$3268`?"
but also "which logical operation produced these bytes, and what optimization
changed them?"

## Investment Rule

Build the map-query telescope as a simple reusable base layer. Avoid spending
heavily on bespoke UI, protocol, or MCP integration until the core query model
has proved useful.

When optimization becomes more ambitious, add provenance beside the map query
layer rather than replacing it. `MapQuery` should remain the physical-address
view; provenance should become the logical-causality view.
